# 有価証券報告書RAGアプリ 機能要件定義

## 1. コンセプト
有価証券報告書を中心とした企業情報をRAGで検索し、**就活に特化した**企業分析ツール。

---

## 2. データソース

| データ | 取得方法 | 更新頻度 |
|---|---|---|
| 有価証券報告書（テキスト） | EDINETからPDFダウンロード → pymupdf4llm でMarkdown変換 | 年1回 |
| 有価証券報告書（財務数値） | EDINET XBRL API | 年1回 |
| 決算短信 | EDINETからPDFダウンロード | 四半期ごと |
| 最新ニュース | Web検索Tool（自動取得） | ページ表示時 |
| IR・企業ホームページ | Web検索Tool | オンデマンド |

---

## 3. ページ構成

```
/companies              ：企業一覧
/companies/{ticker}     ：企業個別ページ（RAGチャット・横断質問も対応）
/companies/compare      ：複数企業比較
/calendar               ：就活カレンダー（スコープ外・後回し）
```

> `/chat`（グローバルチャット）は廃止。企業個別チャットで横断的な質問に対応する。

### 企業個別ページ（/companies/{ticker}）

```
┌─────────────────────────┐
│ 企業概要（事前生成・DB保存）│
│ 売上・利益・事業内容      │
├─────────────────────────┤
│ 財務グラフ               │
│ 売上・営業利益推移等      │
├─────────────────────────┤
│ 最新ニュース（自動取得）  │
│ ・ニュースタイトル+リンク  │
│ [有価証券報告書と照合して分析する]  │ ← オンデマンドでLLM呼び出し
├─────────────────────────┤
│ 💬 RAGチャット           │
│ 詳細を自然言語で質問      │
│ 競合他社との横断質問も可  │
└─────────────────────────┘
```

## 4. データ取得・保存戦略

### 2トラック設計

```
EDINET
└── XBRL ZIP（1ファイルで完結）
      ├── XBRL タグ → xbrl_parser.py → 財務数値 → MongoDB financials
      │                                             ↑ DB Tool経由で正確に取得
      └── iXBRL HTML → Markdown変換
                ↓
            テキストチャンク → Embedding → MongoDB Vector
                                            ↑ RAGで検索
```

**数値系の質問はDB Toolで正確に取得する。** RAGにも財務表を含む全文が入るが、数値クエリはルーターがDB Toolに振るため問題ない。PDFは不要（iXBRLで代替）。

## 5. LLMを呼び出すタイミング

```
有価証券報告書取り込み時（1回だけ）：
　・企業サマリー生成
　・財務データの構造化抽出（XBRLから）
　・リスク要因の整理
　→ 全部MongoDBに保存

ページ表示時（自動・LLM不要）：
　・最新ニュースをWeb検索で自動取得
　・タイトル・概要・リンクを表示するだけ

ユーザ操作時（オンデマンド）：
　・チャットの質問 → RAG + LLM（またはDB Tool）
　・「有価証券報告書と照合して分析する」ボタン → LLM呼び出し
　・ES添削 → RAG + LLM
```

## 6. 機能一覧

### 6.1 企業概要（事前生成・静的）
- 有価証券報告書取り込み時にLLMで一度だけ生成・DB保存
- ページ表示時はDBから取得するだけ
- 種類：
  - 会社全体サマリー
  - 財務ハイライトサマリー
  - 事業セグメント別サマリー
  - リスク要因サマリー

### 6.2 財務グラフ（事前生成・静的）
- 有価証券報告書取り込み時にXBRL APIで取得・DB保存（LLM不要）
- Rechartsでグラフ描画
- 対応グラフ：
  - 売上収益の推移（折れ線）
  - 税引前利益・純利益の推移（折れ線）
  - ROE・自己資本比率の推移（折れ線）
  - セグメント別売上（円・棒グラフ）
  - 複数企業比較（棒グラフ）

> 営業利益はIFRSで標準タグが存在しないため廃止。税引前利益・純利益はIFRS・J-GAAP共通で取得可能。
> UIラベルは `revenue` →「売上（収益）」と表記し、会計基準の差異を吸収する。

### 6.3 最新ニュース（動的・LLM不要）
- ページ表示時にWeb検索で自動取得
- タイトル・概要・リンクのみ表示
- LLMを使わないのでコストゼロ

### 6.4 ニュース × 有価証券報告書照合分析（オンデマンド）
- 「有価証券報告書と照合して分析する」ボタンで任意実行
- Web検索結果 + RAGで取得した有価証券報告書データをLLMに渡す
- 企業の動向・将来予測を生成
- 24時間キャッシュでコスト削減

### 6.5 チャット機能（動的）
- 自然言語で企業情報を質問
- 質問の種類によってToolを自動選択：
  - 数値系（売上・利益率等） → DB接続Tool
  - テキスト系（戦略・リスク等） → RAG
  - サマリー系 → サマリーチャンク
- 「競合のXX社と比べて？」等の横断質問にも対応（複数社のチャンクを動的に検索）
- 短期メモリ（ConversationBufferMemory）で会話の文脈を保持

### 6.6 就活支援機能
- **ES添削**：ユーザーのES文章をRAGクエリとして使い、有価証券報告書の関連チャンクを取得してLLMが添削・フィードバック
- 志望動機の裏付け情報提供
- 面接想定問答の生成

### 6.7 投資判断支援機能
- 財務指標の正確な計算Tool（PER・ROE・営業利益率等）
- 競合他社との財務比較

## 7. RAG設計

### チャンク戦略
```
・PDF → pymupdf4llm → Markdownに変換してからチャンク化
・基本サイズ：500文字
・オーバーラップ：100文字
・セクション単位で区切ることを優先
・セクション情報をメタデータとして付与
```

### メタデータ設計
```json
{
  "text": "...",
  "type": "chunk | summary",
  "section": "事業の状況",
  "ticker": "7203",
  "year": 2024
}
```

### チャンクの種類
```
通常チャンク（type: "chunk"）
　→ 詳細な質問・ES添削に対応

サマリーチャンク（type: "summary"）
　→ 「まとめて」「概要」系の質問に対応
```

### 質問の分岐
```
「まとめて」「概要」系　→ サマリーチャンク
詳細な質問　　　　　　　→ 通常RAG検索
グラフ・数値系　　　　　→ DB Tool（RAGを使わない）
ES添削　　　　　　　　　→ ESをクエリにRAG検索
競合比較　　　　　　　　→ 複数社のチャンクを動的に検索
```

## 8. APIエンドポイント設計

```
/api/chat　　　 → チャット（RAG + LLM or DB Tool）
/api/analyze　 → ニュース×有価証券報告書照合分析（オンデマンド）

/api/es　　　　 → ES添削（RAG + LLM）
```

## 10. DBスケール戦略

```
開発・MVP　　：MongoDB Atlas無料枠（512MB・志望企業10〜20社）
容量超過後　 ：RasPi Self-hosted MongoDB（コストほぼゼロ）
最終目標　　 ：日経225全社対応
```

### Atlas → RasPi移行手順
```bash
# エクスポート
mongodump --uri="mongodb+srv://Atlas接続URL" --out=./backup

# インポート
mongorestore --uri="mongodb://localhost:27017" ./backup

# Vector Indexのみ移行先で再作成が必要
```

## 13. 差別化ポイント

- 就活に特化した企業分析ツール
- 一次情報（有価証券報告書）ベースで精度が高い
- 最新ニュースの自動取得 + 有価証券報告書との照合分析
- 財務データのグラフ可視化
- ESを有価証券報告書で裏付けるES添削機能
- 自分の志望企業に完全特化できる
- 既存サービス（ワンキャリア・マイナビ・四季報）との差別化
