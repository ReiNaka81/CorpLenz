# 開発環境

## 前提ツール

| ツール | 用途 | インストール |
|---|---|---|
| uv | Python パッケージ管理 | `curl -LsSf https://astral.sh/uv/install.sh \| sh` |
| Node.js | フロントエンド実行環境 | [nodejs.org](https://nodejs.org) からインストール（LTS版） |
| git | バージョン管理 | 別途インストール |
| Docker Desktop | コンテナ実行環境 | [docker.com](https://www.docker.com/products/docker-desktop/) からインストール |

## 1. バックエンドのセットアップ

```bash
cd backend

# 仮想環境の作成・有効化
uv venv                        # .venv/ フォルダを作成
source .venv/bin/activate      # 仮想環境を有効化（プロンプトに (.venv) と表示されれば成功）

# 依存パッケージのインストール
uv pip install -r requirements.txt
```

> **VSCode使用時：** 仮想環境作成後、`Cmd+Shift+P` → `Python: Select Interpreter` → `.venv/bin/python` を選択する。

### パッケージを新たに追加するとき

```bash
uv pip install <パッケージ名>
uv pip freeze > requirements.txt  # requirements.txtに反映
```

### 環境変数の設定

```bash
cp .env.example .env
```

`.env` を編集：

```
MONGODB_URI=mongodb+srv://<user>:<password>@cluster.mongodb.net/recruit_app
OPENAI_API_KEY=sk-...
ANTHROPIC_API_KEY=sk-ant-...
TAVILY_API_KEY=tvly-...
LANGCHAIN_API_KEY=ls-...       # LangSmith（開発ロギング用）
LANGCHAIN_TRACING_V2=true
```

## 3. MongoDB Atlas のセットアップ

1. [MongoDB Atlas](https://cloud.mongodb.com) でアカウント作成
2. 無料クラスター（M0）を作成
3. Database User を作成（ユーザー名・パスワードをメモ）
4. Network Access で `0.0.0.0/0`（全IP許可）を設定
5. 接続文字列を `.env` の `MONGODB_URI` に設定

### Vector Search インデックスの作成

Atlas UI の Search タブから以下のインデックスを作成：

```json
{
  "fields": [
    {
      "type": "vector",
      "path": "embedding",
      "numDimensions": 1536,
      "similarity": "cosine"
    },
    {
      "type": "filter",
      "path": "ticker"
    },
    {
      "type": "filter",
      "path": "type"
    }
  ]
}
```

> `numDimensions: 1536` は OpenAI text-embedding-3-small の次元数

## 4. ローカル起動

### Docker を使う場合（推奨）

ルートで1コマンドで起動できる。

```bash
# 初回（イメージビルド＋起動）
docker compose up --build

# 2回目以降
docker compose up
```

| URL | 内容 |
|---|---|
| http://localhost:3000 | フロントエンド |
| http://localhost:8000/docs | バックエンド Swagger UI |

```bash
# 停止
docker compose down

# ログを見る
docker compose logs -f          # 全サービス
docker compose logs -f backend  # バックエンドのみ
```

ソースコードの変更は自動反映される（ホットリロード）。

---

### 手動で起動する場合

ターミナルを2つ開いて起動する。

**ターミナル1：バックエンド**

```bash
cd backend
source .venv/bin/activate
uvicorn api.main:app --reload --port 8000
```

→ http://localhost:8000/docs でSwagger UIが確認できる

**ターミナル2：フロントエンド**

```bash
cd frontend
npm run dev
```

→ http://localhost:3000 でアプリが起動する

## 5. データ取り込み（初回セットアップ）

企業データを取り込むスクリプトを実行する。

```bash
cd backend
source .venv/bin/activate

# 1社取り込み（例：トヨタ = 7203）
python -m ingestion.edinet --ticker 7203 --year 2024

# 実行内容：
#   1. EDINET から XBRL ZIP をダウンロード（PDF不要）
#   2. XBRL タグ → 財務数値 → MongoDB financials に保存
#   3. iXBRL HTML → Markdown → チャンク化 → Embedding → MongoDB chunks に保存
#   4. LLM → サマリー生成 → MongoDB companies に保存
```

### EDINETスクリプトの単体検証

EDINET APIからのPDF・XBRLダウンロードのみを検証する場合：

```bash
cd backend
source .venv/bin/activate

# --ticker: 証券コード4桁、--year: 対象年度
python -m ingestion.edinet --ticker 7203 --year 2024
```

成功すると `backend/data/{会社名}/{year}/` にファイルが保存される：

```
backend/data/
└── トヨタ自動車株式会社/
    └── 2024/
        └── {docID}_xbrl.zip
```

### XBRLパーサーの単体検証

財務数値の抽出結果を確認する場合：

```bash
# 財務数値を出力（5年分）
python ingestion/xbrl_parser.py --company トヨタ --year 2025

# revenueにマッチするタグを全表示（デバッグ）
python ingestion/xbrl_parser.py --company 日立 --year 2025 --debug revenue

# フリーワードでタグ検索
python ingestion/xbrl_parser.py --company 日立 --year 2025 --debug Profit
```

### iXBRL → Markdown変換の単体検証

iXBRL本文をMarkdownに変換した結果を確認する場合：

```bash
# 特定セクションのMarkdownを表示（先頭2000字）
python ingestion/html_parser.py --company 日立 --year 2025 --section 0101010

# 全セクション一覧を表示
python ingestion/html_parser.py --company 日立 --year 2025
```

セクション番号の目安：

| 番号 | 内容 |
|------|------|
| 0101010 | 主要な経営指標等の推移 |
| 0102010 | 事業の内容 |
| 0103010 | 関係会社の状況 |
| 0104010 | 従業員の状況 |
| 0105010 | 経営方針・経営環境 |
| 0106010 | 財務状態・経営成績の分析 |
| 0200010 | 財務諸表 |

## 6. デプロイ

1. [Vercel](https://vercel.com) にGitHubリポジトリを連携

4. Deploy → 以降は `main` ブランチへのpushで自動デプロイ

## 8. コミットメッセージ規約

[Conventional Commits](https://www.conventionalcommits.org/) に準拠する。

```
<type>: <概要（命令形・50字以内）>
```

### type 一覧

| type | 用途 |
|---|---|
| `feat` | 新機能 |
| `fix` | バグ修正 |
| `chore` | ビルド・依存・設定変更（機能に影響しない） |
| `docs` | ドキュメントのみの変更 |
| `refactor` | 動作を変えないコード整理 |
| `test` | テスト追加・修正 |
| `style` | フォーマット等（ロジック変更なし） |

### 具体例

```
feat: EDINET PDFダウンロード機能を追加
feat(rag): サマリーチャンク検索に対応
feat(api): /api/es エンドポイントを追加
fix: Vector Search のフィルター条件が無効になるバグを修正
fix(chat): セッションIDが未設定の場合にエラーが発生する問題を修正
chore: .gitignore にEDINETダウンロードファイルを追加
chore: 依存パッケージを更新
docs: RAG設計書にチャンク戦略を追記
refactor: retriever.py の検索ロジックを関数に切り出し
test: financial Tool の単体テストを追加
```

### ルール

- 概要は日本語OK
- 概要は命令形（「追加した」ではなく「追加」）
- スコープは任意。変更範囲が明確な場合は `feat(rag):` のように付ける

## 8. トラブルシューティング

### MongoDB に接続できない

- Atlas の Network Access に自分のIPが許可されているか確認
- `MONGODB_URI` の形式が正しいか確認（`@` の前後にユーザー名・パスワード）

### Vector Search が動かない

- Atlas のインデックスが `Active` 状態になっているか確認（作成直後は数分かかる）
- `numDimensions` が `1536` になっているか確認

### FastAPI が起動しない

- 仮想環境が有効化されているか確認（`source .venv/bin/activate`）
- `.env` が存在するか確認

### Docker コンテナが起動しない

- Docker Desktop が起動しているか確認
- `backend/.env` が存在するか確認（`cp backend/.env.example backend/.env` して値を設定）
- ポートが競合していないか確認（3000・8000番が他のプロセスで使われていないか）
- イメージを再ビルドする：`docker compose up --build`

## 9. Python モジュール実行の基本

### `python3 script.py` と `python3 -m module` の違い

| | `python3 script.py` | `python3 -m module` |
|---|---|---|
| `sys.path` に追加されるもの | スクリプトのディレクトリ | カレントディレクトリ |
| 相対インポート | 使えない | 使える |
| `__init__.py` | 不要 | サブディレクトリには必要 |

他のファイルから `import` して使うディレクトリには `__init__.py` が必要：

```
backend/
  ingestion/
    __init__.py   ← これがないと main.py から import ingestion.xxx できない
    edinet.py
    chunker.py
```

### `__main__` について

```python
def main():
    ...

if __name__ == "__main__":
    main()  # 直接実行時だけ動く、importしても動かない
```

- **直接実行・`-m` 実行時** → `__name__` が `"__main__"` になる
- **import された時** → `__name__` がモジュール名（例: `"ingestion.chunker"`）になる

モジュールとして使いつつ単体でも実行できるようにする定番パターン。
